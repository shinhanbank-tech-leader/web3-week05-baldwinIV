## 2. EOA(Externally Owned Account)의 한계와 위험성
EOA는 개인키(Private Key)가 곧 계정의 제어권을 의미하는 구조다. 은행 IT 관점에서 이는 관리되지 않는 무기명 채권과 유사한 위험을 내포한다.

* 복구 불가성: 키 분실 시 자산 접근권이 영구적으로 상실된다. 은행의 비밀번호 재발급 프로세스에 익숙한 일반 사용자에게는 수용하기 어려운 리스크다.
* 유연성 부재: 하나의 키로 모든 권한을 행사하므로, 조회 전용이나 이체 한도 설정 같은 세분화된 권한 관리가 불가능하다.
* 서명 복잡도: 모든 트랜잭션마다 가스비(Gas Fee)를 해당 계정의 네이티브 토큰으로 보유해야 하며, 매번 직접 서명해야 하는 번거로움이 있다.

## 3. ERC-4337: 계정 추상화의 등장과 차별점
ERC-4337은 스마트 컨트랙트를 계정처럼 사용하는 기술이다. 기존 EOA가 수동적인 열쇠였다면, ERC-4337은 프로그래밍 가능한 디지털 금고에 가깝다.

* 스마트 계약 기반 제어: 하드웨어 레벨의 개인키가 아닌, 스마트 컨트랙트 코드에 정의된 로직에 따라 트랜잭션이 실행된다.
* Bundler 도입: 사용자의 요청을 모아 처리하는 번들러 덕분에 네이티브 토큰 없이도 트랜잭션 수행이 가능하다.
* Paymaster 활용: 서비스 제공자가 가스비를 대납하거나, 다른 토큰으로 가스비를 지불할 수 있는 유연성을 제공한다.

## 4. 운영 및 통제 측면의 새로운 과제
사용성이 개선됨에 따라 금융 시스템 수준의 통제 메커니즘 구축이 필수적이다.

* 승인 규칙 및 정책: 다중 서명(Multi-sig)이나 이체 한도 설정을 컨트랙트 단에서 구현해야 하며, 이 로직의 취약점은 곧 금융 사고로 직결된다.
* 복구 권한(Social Recovery): 신뢰할 수 있는 제3자나 은행 시스템이 복구 권한을 가질 때, 해당 권한 오남용을 방지할 거버넌스가 필요하다.
* 수수료 부담 주체: Paymaster 운영 시 발생하는 비용을 마케팅 비용으로 처리할지, 수수료 수익 모델로 전환할지에 대한 회계적 정립이 요구된다.
* 책임 소재: 스마트 컨트랙트 오류나 번들러 장애로 인한 거래 지연 발생 시, 은행과 인프라 제공자 간의 SLA 체결 및 책임 범위 획정 가이드라인이 필요하다.
