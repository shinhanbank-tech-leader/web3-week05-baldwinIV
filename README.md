# EOA와 ERC-4337 기반 지갑 비교와 트릴레마를 해결하기 위한 노력

** ERC-4337과 트릴레마 사이의 연결고리를 찾는 게 상당히 어렵네요..
EOA는 실제 쓰기 어려움 -> CA를 써야함 -> 확장성(가스비문제) 트릴레마 -> L2 흐름으로 정리했습니다.


## I. 트릴레마 및 EOA와 CA에 대하여

### 1. 블록체인의 트릴레마

블록체인 네트워크가 직면한 세 가지 핵심 요소인 **확장성(Scalability), 보안성(Security), 탈중앙화(Decentralization)**를 동시에 만족하기 어렵다는 구조적 한계를 트릴레마라고 한다.

예를 들어 

이더리움은 보안성과 탈중앙화를 얻은 대신, 확장성(모든 노드가 데이터를 검증해야 하여 속도가 느리고 사용자 증가시 가스비 폭등)을 어느정도 포기하였다.
솔라나는 확장성과 보안성을 얻은대신 탈중앙화를 희생하였다. (초고속 처리를 위해 노드에 고성능 CPU를 요구)

현재로서는 메인넷 단독으로는 트릴레마를 해결하기가 요원해보인다.

### 2. ERC-4337 (CA) 의 의의
기존 블록체인 계정은 단순한 입출금 가능한 계좌인 EOA만 존재하였다. 하지만 개인키 분실 시 자산 접근권이 영구 상실되는 리스크는 우리와 같은 제도권 금융에서는 사용하기에 애로사항이 존재한다. 이에 프로그래밍 가능한 **CA(Contract Account)**를 지갑처럼 사용하는 '계정추상화'가 대안으로 등장한다.

# II. CA의 확장성 문제와 L2를 통한 해결

## 1. ERC-4337(CA)가 좋은데.. 왜??

가스비 부담이 있다. CA의 수많은 기능들 (페이마스터, 계정복구 등등)은 모두 메인넷 상의 대량의 가스비 부담을 갖게 된다. 트릴레마중 '확장성'에 이슈가 있는 것이다.

## 1. 트릴레마의 하드웨어적 해결 : Layer 2
트릴레마 중 확장성을 해결하기 위해 등장한 Layer 2(L2)는 메인넷(L1) 외부에서 트랜잭션을 처리하고 결과만 기록하는 방식이다. 이를 통해 연산 비용을 대폭 절감하고 처리 속도를 높일 수 있다. (낙관적 롤업 혹은 영지식 롤업 등을 통해 거래를 롤업하여 처리)

## 2. L2와 계정추상화
L2가 가스비 문제를 해결하게 되어, 계정추상화(AA)를 CA를 통해 사용할 수 있게 되었다.
--> L2의 저렴한 가스비를 기반으로 AA의 여러 로직을 실행할 수 있게 됨!

 * EOA의 문제 (별첨)
 ---
  -  **복구 불가:** 키 분실 시 자산 접근 권한이 영구 소멸되어 고객 민원 대응 불가.
  - **일방적 권한:** '전부 아니면 전무(All or Nothing)' 구조로 인해 이체 한도 설정 등 세부 통제 불가능.
   - **사용성 저하:** 가스비 지불을 위한 별도 토큰 보유 및 매 단계 수동 서명 필요.
## 3. 은행 시스템 vs ERC-4337 비교
은행 중앙 서버가 관리하는 것을.. 블록체인상에서는 ERC-4337상 비즈니스 로직이 대신한다.

* 인증 : 은행 서버 대조 vs 온체인 스마트 컨트랙트 코드 검증
* 승인 : 내부 DB 권한 제어 vs 컨트랙트 내 조건문(If-Then) 자동 실행
* 계정 관리 : 관리자가 DB 직접 수정 vs 가디언(Guardian) 로직을 통한 권한 이전
* 수수료 : 시스템 비용으로 흡수 vs Paymaster를 통한 유연한 대납 구조 설계

# III. 정리
## 1. 당행에서는 어떤 구조로 받아들일 수 있을까?
은행 IT 개발자 관점에서 가장 현실적인 모델은 **'하이브리드 비수탁 모델'**이다.
* **Managed Guardian:** 고객이 개인키를 갖되, 은행이 컨트랙트 가디언으로 참여하여 분실 시 복구 권한을 보증함.
* **Corporate Paymaster:** 대출 고객에게 가스비 허들을 제거하기 위해 은행이 가스비를 전액 대납하고, 마케팅 비용으로 처리하거나 원화 수수료에 산입함.
* **Policy-Driven Wallet:** 대출 실행 및 상환 시점에만 지갑의 특정 기능을 활성화하는 로직을 심어 자산 건전성을 관리함.
## 2. 금융회사 IT 관점의 현실적 수용 구조
은행권에서 가장 현실적으로 받아들일 수 있는 구조는 하이브리드 비수탁 모델입니다.
* Managed Guardian : 고객이 키를 소유하되 은행이 복구 가디언으로 참여하여 제도권 수준의 보안 보증
* Corporate Paymaster : 대출 고객의 UX를 위해 가스비 허들을 제거하고 은행이 비용을 일괄 정산
* Policy-Driven Wallet : 대출 담보 관리 및 상환 시점에만 특정 기능이 활성화되는 정책 기반 지갑 구현
## 3. 결론
EOA를 통해서는 기존 금융과 같이 각 고객에 대한 세밀한 비즈니스적 관리가 불가하니 CA기반의 ERC-4337지갑을 사용하는 방향으로 가야하나, 확장성 문제가 있다.
그렇기 때문에 L2를 통해 ERC-4337의 확장성 문제를 해결할 수 있다.
은행에서는 블록체인 거래에 있어 CA를 사용하며, 상술한 하이브리드 비수탁 방식으로 블록체인 서비스를 제공할 수 있을 것이다.
